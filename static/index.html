PlantBox

Open Source Tools for Plant Bioinformatics

PlantBox is a lightweight interface for exploring plant genomic data hosted on Ensembl at EMBL-EBI. It queries the most recent EMBL-EBI SQL databases and REST APIs directly, offering an efficient alternative to manual queries through the Ensembl browser. PlantBox also provides additional features not available through Ensembl.

The open source nature of bioinformatics data provides a wealth of opportunity to develop new features and tools to streamline research and scientific enquiry. If you have suggestions or feedback, please get in touch below.

Tools

Variant Explorer & Primer Design

Explore all genes associated with wheat variants (Cadenza and Kronos). Supports wildcard search — for example, TraesCS3A02 retrieves all genes on Chr 3A. KASP markers are returned where available, generated by the BBSRC Nottingham Wheat Research Centre.

Includes an integrated KASP & PCR Primer Designer for variants — click any result row to design KASP or standard PCR primers directly from variant data. Returns up to 5 ranked primer pairs with Tm, GC%, hairpin, and self-complementarity stats.

How it works: Queries the variation and core MySQL databases concurrently, then left-joins results on variant name. For primer design, fetches flanking genomic sequence from the Ensembl REST API and runs Primer3 to design allele-specific (KASP) or standard forward/reverse primer pairs.

Note: Results capped at 300 rows for speed. Primer designer is embedded — click any result row to activate. Tool restricted to Wheat only.

Sequence Retrieval

Rapid retrieval of CDS, genomic, protein, and cDNA sequences for up to 50 gene IDs at a time. Results are returned as FASTA and can be downloaded directly.

How it works: Calls the Ensembl Plants REST API for each ID. All sequence versions are returned.

Note: Up to 50 gene IDs per query.

Annotate DNA Sequence

Annotates intron and exon boundaries of a given gene across canonical and non-canonical transcripts. Renders an interactive colour-coded sequence view with per-segment tooltips.

How it works: Two concurrent REST calls fetch gene metadata and the full genomic sequence; exon coordinates are mapped onto the sequence to produce segments.

Note: One gene ID at a time; supports any Ensembl Plants gene.

Annotate Amino Acid Sequence

Annotates protein residues with domain databases (Pfam, Panther, Prosite, Prints, Superfamily). Provides interactive colour-coded domain coverage with overlap resolution.

How it works: Queries the species core database for protein features and the canonical transcript, then fetches the protein sequence from the REST API.

Note: One gene ID at a time; select species and domain databases before querying. Tool restricted to Wheat, Barley, Rice, Arabidopsis and Maize.

Pull Homologous Sequences and Protein Pairwise Alignment

Returns all orthologues, homoeologues, or paralogues of a gene, including sequences not shown in the Ensembl browser. An interactive pairwise alignment viewer reconstructs protein alignments from CIGAR strings client-side, with a summary statistics table showing identity %, matches, mismatches, gaps, and residue composition.

How it works: Queries the Ensembl Compara Plants database; alignments are reconstructed from CIGAR strings in the browser — no alignment algorithm is executed server-side.

Note: One gene ID at a time; CSV/Excel downloads include full sequences and CIGAR lines. The .txt alignment export includes the full statistics table.

Promoter & Cis-Regulatory Element Annotator

Fetch an upstream promoter window and scan for 16 plant cis-regulatory elements (light, ABA, heat, defense, hormone, and circadian motifs). Two views: colour-coded sequence and graphical SVG overview showing motif positions on both strands.

How it works: Fetches gene coordinates from the Ensembl REST API, retrieves the upstream genomic window, then scans both strands for IUPAC motif consensus sequences using regex matching.

Note: One gene ID at a time; upstream window configurable from 200 to 5000 bp. Supports any Ensembl Plants gene.

        Component Reference
      


      
      

        
Variant Explorer & KASP Primer Design

        

          

            
              

                Category
                Details
              

            
            
              

                API Calls
                
                  
POST /api/query — variant + KASP marker search. Accepts gene_id or transcript_id + optional variant_id.

                  
POST /api/primers — KASP or PCR primer design for a selected variant. Accepts variant_name, chromosome, position, allele_string, flanking_bp (50–500), num_pairs (1–10), primer_type.

                  
GET /api/download/csv/{token} and /excel/{token} — stream cached rows as CSV or Excel.

                		
              

              

                Data Flow
                
                  
/api/query: fetch_and_join() fires run_variant_query() and run_marker_query() concurrently via asyncio.gather, then left-joins on variant_name. _fetch_strandedness() makes a batched POST /lookup/id to the Ensembl REST API and inserts a Strandedness column. Results serialised and stored in _result_cache under a UUID token.

                  
/api/primers: fetches flanking sequence from GET /sequence/region/triticum_aestivum/{chr}:{s}..{e}, then calls primer3.design_primers() inside asyncio.to_thread() (blocking call offloaded). ALT primer thermodynamics computed with calcTm, calcHairpin, calcHomodimer. No caching — direct response.

                		
              

              

                Databases & Caches
                
                  
variation_pool → triticum_aestivum_variation_* on mysql-eg-publicsql.ebi.ac.uk:4157 — variant consequence, SIFT, allele strings.

                  
core_pool → triticum_aestivum_core_* — BBSRC WRC KASP marker left/right primers.

                  
_result_cache — OrderedDict, TTL 30 min, max 100 entries. Token lookup is O(1); download endpoints reuse it without re-querying.

                		
              

            
          

        

      


      
      

        
Annotate DNA Sequence

        

          

            
              

                Category
                Details
              

            
            
              

                API Calls
                
                  
POST /api/annotate — accepts { gene_id } (validated via validate_input()).

                  
Returns AnnotateResponse: gene metadata, full genomic sequence, per-transcript segment lists (exon/intron), and the canonical transcript ID.

                		
              

              

                Data Flow
                
                  
Two concurrent httpx requests via asyncio.gather:

                  
① GET /lookup/id/{gene_id}?expand=1 — gene metadata + transcript/exon coordinates.

                  
② GET /sequence/id/{gene_id}?type=genomic — full genomic sequence string.

                  
_compute_segments() maps each transcript's exon coordinates onto the genomic sequence to produce {type, seq_start, seq_end, number} segments. Canonical transcript sorted first. Result stored in _annotate_cache.

                		
              

              

                Databases & Caches
                
                  
No MySQL query — purely REST-based.

                  
_annotate_cache — same OrderedDict policy (TTL 30 min, max 100). Keyed by UUID token returned to the client.

                  
Frontend renders colour-coded spans (odd exon = blue, even exon = sky, intron/flank = grey) with per-segment hover tooltips showing type, number, and length.

                		
              

            
          

        

      


      
      

        
Annotate Amino Acid Sequence

        

          

            
              

                Category
                Details
              

            
            
              

                API Calls
                
                  
POST /api/annotate_protein — accepts { gene_id, species, domains[] }. Species must be in SUPPORTED_SPECIES; at least one domain database required.

                  
Returns ProteinAnnotateResponse: protein sequence + domain feature list (type, hit_name, description, start, end).

                		
              

              

                Data Flow
                
                  
_get_species_core_pool() lazily creates (or retrieves from _species_core_pools) an aiomysql pool for {species}_core_*.

                  
Two concurrent DB calls: fetch_canonical_transcript_id() and fetch_protein_domains() query gene → transcript → translation → protein_feature, filtered to the canonical transcript and to selected domain prefixes (PF, PTHR, PS, PR, SSF).

                  
Then GET /sequence/id/{tx_id}?type=protein fetches the protein sequence using the canonical transcript ID (avoids Ensembl's "multiple sequences" error).

                  
Result cached in _prot_annotate_cache.

                		
              

              

                Databases & Caches
                
                  
species_core_pool(s) → {species}_core_* — one pool per species, created lazily and persisted for the server lifetime. Closed on shutdown.

                  
_species_core_pools — dict acting as a persistent pool registry (not TTL-evicted).

                  
_prot_annotate_cache — TTL 30 min, max 100. Client-side paIntervals getter decomposes the sequence at domain boundaries, priority-sorts overlaps (Pfam > Panther > Prosite > Prints > Superfamily), and renders colour-coded spans.

                		
              

            
          

        

      


      
      

        
Pull Homologous Sequences & Protein Pairwise Alignment

        

          

            
              

                Category
                Details
              

            
            
              

                API Calls
                
                  
POST /api/homology — accepts { gene_id, homology_type }. Valid types: orthologues, homoeologues, paralogues (each maps to multiple SQL description values).

                  
GET /api/download/csv/{token} and /excel/{token} — all 11 columns including full sequences and CIGAR lines. Filename: {gene_id}_homology.csv.

                		
              

              

                Data Flow
                
                  
fetch_homologs() builds a parameterised OR-chain of h.description = %s clauses and queries homology → homology_member → gene_member / genome_db / seq_member / sequence, joined twice (once for the query gene, once for the homologue). Bytes in sequences and CIGAR lines are decoded UTF-8 (latin1 charset edge case).

                  
Results cached in _result_cache via _cache_store() with filename="{gene_id}_homology". The download endpoints reuse this entry and derive the filename from the cache.

                		
              

              

                Databases & Caches
                
                  
compara_pool → ensembl_compara_plants_* — Ensembl Compara database containing homology relationships, CIGAR alignments, and member sequences.

                  
_result_cache — shared with the Variant Explorer (keyed independently by UUID token).

                  
Frontend reconstructs pairwise alignments client-side from CIGAR strings (M = consume residue, D = insert gap). Alignment wraps to container width via ResizeObserver. A summary statistics table below the alignment blocks shows identity %, matches, mismatches, query/homolog gap counts, sequence lengths, aligned length, and residue composition — all computed client-side from the aligned sequences. These stats are also appended to the .txt export.

                		
              

            
          

        

      


      
      

        
Sequence Retrieval

        

          

            
              

                Category
                Details
              

            
            
              

                API Calls
                
                  
POST /api/sequences — accepts a list of gene IDs and a sequence type (CDS, genomic, protein, cdna). Up to 50 IDs.

                  
GET /api/download/fasta/{token} — streams cached FASTA text.

                		
              

              

                Data Flow
                
                  
Calls POST /sequence/id on the Ensembl Plants REST API as a batch request. All sequence versions are returned. Result FASTA text stored in _fasta_cache under a UUID token.

                		
              

              

                Databases & Caches
                
                  
No MySQL query — purely REST-based.

                  
_fasta_cache — OrderedDict, TTL 30 min, max 100 entries. The download endpoint does an O(1) token lookup and streams the stored FASTA string.

                		
              

            
          

        

      


      
      

        
Promoter & Cis-Regulatory Element Annotator

        

          

            
              

                Category
                Details
              

            
            
              

                API Calls
                
                  POST /api/promoter — accepts { gene_id, upstream_bp } where upstream_bp is clamped to 200–5000 (default 2000). Returns gene metadata, the promoter sequence, and all cis-regulatory element hits with positions and strand.
                
              

              

                Data Flow
                
                  
Two sequential Ensembl REST calls: GET /lookup/id/{gene_id}?expand=0 to obtain chromosome, start/end coordinates, strand, and species; then GET /sequence/region/{species}/{chr}:{start}..{end} to fetch the upstream window. For reverse-strand genes the sequence is reverse-complemented so index 0 is always the distal end and the last position is the TSS.

                  
_scan_promoter_motifs() iterates all 16 entries in PROMOTER_MOTIFS, compiles each IUPAC consensus into a Python regex via _iupac_to_regex() (M=[AC], Y=[CT], R=[AG], S=[GC], N=[ACGT], W=[AT]), and searches both the forward strand and its reverse complement. Reverse-strand match coordinates are remapped to forward-strand space. Hits are returned sorted by start position and stored in _promoter_cache under a UUID token.

                		
              

              

                Motif Library
                
                  
16 plant CREs across 12 regulatory categories:

                  

                    Core Promoter — TATA box, CAAT box
                    Light — G-box, I-box, GATA motif, Box-4
                    ABA/Drought — ABRE
                    Cold/Drought — DRE/CRT
                    Heat Stress — HSE
                    Defense — W-box
                    JA/SA — TGACG motif
                    Salicylic Acid — TCA element
                    Auxin — AuxRE
                    Ethylene — GCC box
                    MYB — MBS
                    Circadian — Evening element
                  

                		
              

              

                Databases & Caches
                
                  
No MySQL query — purely REST-based and in-process regex scanning.

                  
_promoter_cache — OrderedDict, TTL 30 min, max 100 entries. Stores the full PromoterResponse dict under a UUID token.

                		
              

            
          

        

      


      
      

        
In-Memory Cache Layer

        

          

            
              

                Aspect
                Details
              

            
            
              

                Storage
                All result caches (_result_cache, _fasta_cache, _annotate_cache, _prot_annotate_cache, _promoter_cache) are Python OrderedDict instances stored on app.state. Each entry is a tuple (timestamp, payload).
              

              

                Eviction
                On every write, entries older than 30 minutes are expired first; if the dict still exceeds 100 entries, the oldest entry (FIFO) is removed.
              

              

                Token access
                Every POST endpoint returns a UUID token. Download endpoints do an O(1) dict lookup — no database re-query is ever needed to serve a download.
              

              

                Process isolation
                Caches are in-process RAM only. uvicorn --workers 1 is mandatory — multiple workers would each have independent caches, causing token misses.
              

              

                Species pools
                _species_core_pools is a separate persistent dict (no TTL) holding one aiomysql.Pool per species string. Pools are created lazily on first use and closed during the FastAPI shutdown lifecycle event.
              

            
          

        

      


      
      

        
EBI MySQL Connection Pools

        

          
All pools connect to mysql-eg-publicsql.ebi.ac.uk:4157 as user ensro (no password). At startup, discover_latest_db() lists all databases matching each prefix and picks the one with the highest numeric suffix (e.g. 110_1 > 99_1).

          

            
              

                Pool
                Details
              

            
            
              

                variation_pool
                Database: triticum_aestivum_variation_*. Tables: variation_feature, transcript_variation, allele. Used by run_variant_query().
              

              

                core_pool
                Database: triticum_aestivum_core_*. Tables: marker, marker_feature, seq_region. Used by run_marker_query() to retrieve BBSRC WRC KASP primer sequences.
              

              

                compara_pool
                Database: ensembl_compara_plants_*. Tables: homology, homology_member, gene_member, genome_db, seq_member, sequence. Used by fetch_homologs().
              

              

                species_core_pool(s)
                Database pattern: {species}_core_*. One pool per species (wheat, barley, rice, arabidopsis, maize). Tables: gene, transcript, translation, protein_feature. Used by fetch_canonical_transcript_id() and fetch_protein_domains().
              

            
          

        

      


      
      

        
Ensembl REST API Calls

        

          
All REST calls go to https://rest.ensembl.org via httpx.AsyncClient. Seven distinct endpoints are used:

          

            
              

                Endpoint
                Description
              

            
            
              

                POST /lookup/id
                Batch strandedness lookup called by _fetch_strandedness() after /api/query. Accepts a JSON body of transcript IDs; returns strand and chromosome metadata inserted as the Strandedness column in query results.
              

              

                GET /lookup/id/{id}?expand=1
                Gene metadata including transcript and exon coordinate arrays. Called concurrently with the genomic sequence endpoint by /api/annotate.
              

              

                GET /sequence/id/{id}?type=genomic
                Full genomic sequence string for a gene. Called concurrently with the lookup endpoint by /api/annotate. _compute_segments() maps exon coordinates onto this string.
              

              

                POST /sequence/id
                Batch FASTA retrieval for up to 50 gene IDs at a time. Called by /api/sequences. Response is stored verbatim in _fasta_cache.
              

              

                GET /sequence/id/{tx_id}?type=protein
                Protein sequence for the canonical transcript (queried by transcript ID, not gene ID, to avoid Ensembl's "multiple sequences" error). Called by /api/annotate_protein.
              

              

                GET /sequence/region/
triticum_aestivum/{chr}:{s}..{e}		
                Flanking genomic region around a SNP. Called by /api/primers; the returned sequence is passed to primer3.design_primers() with SEQUENCE_FORCE_LEFT_END (KASP) or SEQUENCE_TARGET (PCR) set at the SNP offset.
              

              

                GET /sequence/region/
{species}/{chr}:{s}..{e}		
                Upstream promoter window for any supported species. Called by /api/promoter after the gene lookup; species is taken from the lookup response so any Ensembl Plants genome is supported. The sequence is reverse-complemented for minus-strand genes before motif scanning.
